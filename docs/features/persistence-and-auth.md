# Persistence & Auth

Design doc for the data layer (`@cup/db`), magic-link authentication, and the server-side
authorization layer. Implements functional-spec §5, §8, §10, §13 and technical-spec §6, §7, §11, §14.

## Packages & layout

- **`@cup/db`** — Drizzle schema (all functional-spec §10 tables), a generic client factory, the
  committed SQL migration, typed repositories, and the pglite test harness. Consumed by `apps/web`
  and (later) the sync script.
- **`apps/web`** — Next.js App Router app. `shared/` holds env validation, the server-only DB
  singleton, the authorization policy layer, and the logger; `features/auth/` holds the Auth.js
  wiring, session helper, and the display-name settings action + minimal UI.

```
packages/db/src/
  client.ts                 # Db<TSchema>; createDb(connStr, schema?) / createPgliteDb(schema?)
  schema/{auth,tournament,pools,predictions,scores,rate-limits}.ts
  migrations/               # single committed initial migration (drizzle-kit)
  repositories/{users,pools,members,kicks,scores,rate-limits}.ts
  testing/{make-test-db,fixtures}.ts   # @cup/db/testing
apps/web/src/
  shared/{env,db}.ts  shared/observability/logger.ts  shared/authz/{errors,actor,policy}.ts
  features/auth/{auth,auth.config,create-user,display-name,email-provider,session,actions}.ts
  app/{page,settings/page}.tsx  app/api/auth/[...nextauth]/route.ts  middleware.ts
```

## Data model

Every functional-spec §10 table is modelled in Drizzle with real FKs and constraints. Highlights:

- **Auth** (`users`, `accounts`, `sessions`, `verificationTokens`) match the `@auth/drizzle-adapter`
  Postgres schema exactly; `users.display_name` is an app addition (NOT NULL, default `''`, set on
  first sign-in). Adapter columns keep their required camelCase names.
- **Tournament/results** tables use per-tournament natural-key composite PKs (`teams`, `stage_groups`,
  `matches`, …). jsonb columns are typed via `.$type<Scoring>()` / `.$type<ScoreBreakdown>()`.
- **Pools/predictions** enforce the domain invariants at the DB: unique `(pool_id, user_id)` on
  `pool_members`, `predictions`, and `scores`; composite PKs on the prediction child tables and
  `pool_kicks` (so "kicked → can't silently rejoin" is airtight); unique `invite_token_hash`.

The single migration is generated by drizzle-kit and applied verbatim by the test harness — schema
and migration cannot drift.

## Authorization (the RLS replacement)

There is no Postgres RLS. The browser never touches the DB; all access is server-side and funnels
through repositories, gated by `shared/authz/policy.ts` (functional-spec §6.5/§8.3/§8.5):

- **Lock:** a member may edit their own card only while `now < firstKickoff` (authoritative server
  clock, injected). Membership is checked **before** the lock so a non/kicked member always gets
  `ForbiddenError`, never a `LockedError` that would leak lock state.
- **Owner override:** the pool owner may edit any member's card at any time, including after lock.
- **Visibility:** own card always; owner any time; other members' cards only after lock; the audit
  log is readable by all members.

Errors are a small typed hierarchy (`ForbiddenError`, `LockedError`, `NotFoundError`) with clear,
actionable messages. The clock is injected everywhere — policy code never reads the wall clock.

## Authentication

Auth.js v5 with the Drizzle adapter and **database sessions**. Magic links are sent via **Resend**
behind an injectable `EmailSender` seam (the only mocked boundary in tests). On user creation,
`applyDerivedDisplayName` sets `display_name` from the email local-part (non-fatal if it fails — the
user can edit it in settings). `getCurrentActor()` maps the session to the authz `Actor`.

## Testing

Test-diamond, integration-first: repositories, the authorization matrix, and the rate limiter are
tested against a **real in-memory Postgres (pglite)** via `makeTestDb()` (fresh DB + applied
migrations per test) — no DB mocks. Pure units (env parsing, `deriveDisplayName`, the email seam,
scoring engine) are unit-tested. The injected clock and the Resend sender are the only mocked seams.

## Environment

Validated at startup by `shared/env.ts` (zod), failing fast and naming every missing/invalid var.
See `.env.example`:

- `DATABASE_URL` — any Postgres (Neon recommended).
- `AUTH_SECRET` — Auth.js session/JWT secret.
- `AUTH_URL` — app base URL.
- `RESEND_API_KEY` — magic-link email delivery.

No secrets or PII are logged (emails are reduced to their domain; ids are opaque).

## Deferred

- **Browser e2e** of the full magic-link flow (request → click → signed in) → Playwright pass.
- **Pool/prediction services** (create/join/kick/submit-card/owner-edit/import/rescore) and **styled
  UI** → the feature-slice plans (4–5); this layer provides the repositories + authz primitives they
  build on.
- **Test DB performance**: `makeTestDb()` applies the full migration per test. Trigger: when the
  pglite suite exceeds ~120s wall time, switch to a per-module migrated DB + transaction rollbacks.
